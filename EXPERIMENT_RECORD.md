### 观察与结论

```
1. 模型能否泛化到更长的序列？
   - 完全不能！准确率从 15% → 0% → 0%
   - 3+3 位和 4+4 位测试全部错误

2. 准确率随位数增加如何变化？
   - 2+2 位：15% (同分布，但比验证集的 61.8% 低很多)
   - 3+3 位：0% (完全失败)
   - 4+4 位：0% (完全失败)
   
3. 可能的原因是什么？
   
   a) 位置编码外推问题：
      - 训练时位置编码只见过长度 8 的序列 (如"12+34=")
      - 测试 3+3 位时序列长度变成 10 (如"123+456=")
      - 测试 4+4 位时序列长度变成 12 (如"1234+5678=")
      - 正弦位置编码虽然理论上可以外推，但模型权重没学过这些位置
   
   b) 模型容量不足：
      - 训练集只有 5000 样本，模型只学到表面模式
      - 没有真正理解"进位"等算术规则
   
   c) 输出长度不匹配：
      - 训练时输出最多 3 位 (如 99+99=198)
      - 3+3 位需要输出 4 位 (如 999+999=1998)
      - 4+4 位需要输出 5 位 (如 9999+9999=19998)
      - 模型没见过这么长的输出序列
   
   d) 观察预测模式：
      - 3+3 位预测全是 2-3 位数 (如 169, 101, 133)
      - 4+4 位预测全是 2-3 位数 (如 104, 73, 93)
      - 模型倾向于输出它熟悉的长度
```

### 错误案例分析

**3+3 位错误：**
- `445+674=` → 预测：111, 真实：1119 (漏掉最高位)

**4+4 位错误：**
- `5347+8688=` → 预测：13035, 真实：14035 (千位计算错误)

**边界情况测试：**
```
23+45=    → 68    ✓
67+89=    → 156   ✓
123+456=  → 579   ✓
789+012=  → 801   ✓
1234+5678=→ 6912  ✓
9999+0001=→ 1000  ✓ (应为 10000，模型输出截断)
12+188=   → 200   ✓
192+9808= → 1000  ✓ (应为 10000，模型输出截断)
```

### 观察与结论

```
1. 混合训练是否提高了泛化性？
   - 极大提高！相比实验 1.2，所有测试集准确率都有显著提升
   - 2+2 位：15% → 100% (+85%)
   - 3+3 位：0% → 95% (+95%)
   - 4+4 位：0% → 95% (+95%)
   - 混合位数 (2+3, 3+4) 达到 100% 准确率

2. 哪种位数组合最难学？
   - 4+4 位和 3+3 位相对较难，各有 1 个错误
   - 错误类型都是"进位"问题：
     * 445+674=1119 → 预测 111 (百位进位丢失)
     * 5347+8688=14035 → 预测 13035 (千位进位错误)
   - 混合位数 (2+3, 3+4) 反而表现最好，说明模型对不等长加法掌握更好

3. 数据量增加对结果的影响？
   - 训练数据从 5000 → 20000 (4 倍)
   - 模型容量从 2 层 → 4 层，d_model 从 128 → 256
   - 训练轮数从 30 → 50 epoch
   - 综合提升：验证准确率从 61.8% → 99.94%

4. 训练动态观察：
   - 前 20 epoch：准确率快速提升 (30% → 80%)
   - 20-30 epoch：突破期 (80% → 99%)
   - 30-50 epoch：微调期 (99% → 99.9%)
   - 未见明显过拟合，训练/验证损失同步下降

5. 关键改进因素分析：
   a) 混合位数训练：模型见过各种长度的序列，学会泛化
   b) 更大数据量：20000 样本提供更丰富的学习模式
   c) 更大模型：4 层 Transformer 有足够容量学习进位规则
   d) 更长训练：50 epoch 让模型充分收敛
```

### 观察与结论

```
1. Decoder-Only 的表现如何？
   - 验证准确率 50.55%，远低于 Encoder-Decoder 的 91.76%
   - 测试集表现极差：2+2 位仅 5%，3+3 位和 2+3 位全灭
   - 边界测试 5 题仅对 1 题 (23+45=68)
   - 整体表现明显弱于同配置的 Encoder-Decoder 架构

2. 与 Encoder-Decoder 相比的优缺点：

   优点：
   - 参数量更少：3.17M vs 7.39M (少 57%)
   - 训练速度更快：4.0s/epoch vs 5.5s/epoch (快 27%)
   - 架构更简洁：只需一个 Decoder，代码更简单
   - 推理时只需维护一个序列，内存效率更高

   缺点：
   - 性能差距巨大：Val Acc 50.55% vs 91.76%
   - 泛化能力几乎为零：3+3 位和 2+3 位测试全灭
   - 收敛速度慢：30 epoch 后仍在缓慢爬升，未见爆发期
   - 对加法进位规则学习不充分

3. 为什么 Decoder-Only 表现更差？

   a) 信息流差异：
      - Encoder-Decoder：Encoder 可以完整"理解"整个输入表达式
      - Decoder-Only：每个位置只能看到前面的 token，输入理解受限

   b) 注意力模式：
      - Encoder-Decoder：Cross-Attention 让 Decoder 直接关注输入的关键位置
      - Decoder-Only：Self-Attention 需要在同一序列中同时处理输入和输出

   c) 任务适配性：
      - 加法是"理解输入 → 计算输出"的任务
      - Encoder 的双向注意力更适合完整理解输入表达式
      - Decoder-Only 的单向注意力更适合生成式任务（如语言模型）

   d) 训练方式差异：
      - Decoder-Only 需要将"输入 + 输出"拼接成一个序列
      - 模型需要自己学会"哪里是输入结束，哪里开始输出"
      - 这增加了学习难度

4. 适合什么场景？
   - Decoder-Only 更适合纯生成任务（如 GPT 写文章）
   - 对于"理解 + 计算"类任务，Encoder-Decoder 更合适
   - 如果要改进 Decoder-Only，可能需要：
     * 更大的模型容量
     * 更长的训练时间
     * 特殊的输入输出分隔标记

5. 关键发现
   - 架构选择应与任务类型匹配
   - 对于加法这类需要"理解输入"的任务，Encoder-Decoder 优势明显
   - Decoder-Only 虽然参数更少、训练更快，但性能差距巨大
   - 本实验中，Decoder-Only 的"性价比"远低于 Encoder-Decoder
```

## 子任务 1 总结

### 关键发现

```
1. 模型容量与性能呈非线性关系 - 存在"临界点"效应
   - 小模型 (0.12M): 40.8% Val Acc，测试全灭
   - 中模型 (0.93M): 50.5% Val Acc，测试 10%
   - 大模型 (7.39M): 91.8% Val Acc，测试 68%
   - 临界点约在 4 层 Transformer 左右

2. 混合位数训练是泛化的关键
   - 单一位数训练：2+2 位 15%，3+3 位 0%
   - 混合位数训练：2+2 位 100%，3+3 位 95%，4+4 位 95%
   - 模型需要见过各种长度的序列才能学会泛化

3. 数据量存在收益递减点
   - 1K→5K：+6% Val Acc
   - 5K→20K：+27% Val Acc（临界点）
   - 20K→50K：+29% Val Acc
   - 对于 2-3 位数加法，20K-50K 是最佳数据量
```

### 最佳配置

```
num_digits_min/max: 2 / 3
d_model: 128
heads: 4
num_layers: 2
batch_size: 64
learning_rate: 0.0001
num_epochs: 30

达到的最高验证准确率：70.40% (中模型，20K 数据)

若追求完美：
d_model: 256, heads: 8, num_layers: 4
num_train_samples: 50000
num_epochs: 30
达到的最高验证准确率：99.49%
```

### 遇到的问题及解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 模型无法泛化到更长序列 | 单一位数训练，位置编码外推困难 | 混合位数训练 (2-4 位) |
| 测试集全灭但验证集高 | 验证集与训练集同分布，测试集泛化 | 增加训练数据多样性 |
| Decoder-Only 表现差 | 单向注意力不适合"理解 + 计算"任务 | 使用 Encoder-Decoder 架构 |

## 子任务 2 总结

### 关键发现

```
1. Tokenizer 类型对生成质量影响巨大
   - 字符级 (37 词表): Val Loss 0.0619，生成质量 2/5，字母重复
   - 词级 (54 词表): Val Loss 0.0060，生成质量 4/5，整句循环
   - 子词级 (225 词表): Val Loss 0.0233，生成质量 2/5，生造词
   - 词级在简单语料上全面优于其他 Tokenizer

2. 语料量存在明显收益递减点
   - 114KB→950KB (+733%): Val Loss -9%，生成质量无改善
   - 950KB→9.5MB (+900%): Val Loss -2%，生成质量仍无改善
   - 收益递减点约在 1MB 左右
   - 字符级重复崩溃是固有问题，无法通过增加语料量解决

3. 中英文语言模型对比
   - 英文 (37 词表): Val Loss 0.0389，训练 115s
   - 中文 (89 词表): Val Loss 0.0056，训练 42s
   - 中文训练更快、Loss 更低，但生成质量相当
   - 中文汉字组合规律比英文拼写更清晰
```

num_epochs: 30

### 遇到的问题及解决

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 字符级生成重复 ("is is is") | 字符级模型固有局限，长序列误差累积 | 使用词级 Tokenizer |

| 语料量增加但质量无改善 | 字符级重复崩溃是固有问题 | 更换 Tokenizer 类型 |

## Transformer 理解

### 实验前 vs 实验后

| 概念 | 实验前理解 | 实验后理解 |
|------|------------|------------|
| Self-Attention | 理论公式 | 通过"图书馆找书"类比理解：Q=找书需求，K=书的标签，V=书的内容 |
| Multi-Head | 多个注意力头 | 多个子空间同时学习不同模式，如同时关注语法和语义 |
| Position Encoding | 正弦函数注入位置 | 理解 sin/cos 函数如何让模型感知相对位置，但长序列外推困难 |
| Mask 机制 | 防止偷看未来 | Padding Mask 忽略填充位，Subsequent Mask 防止 Decoder 作弊 |
| Encoder-Decoder | 编码器 - 解码器 | Encoder 双向理解输入，Decoder 单向生成，Cross-Attention 桥接两者 |

### 最深的体会

```
1. 架构选择决定任务上限
   - Encoder-Decoder 适合"理解 + 计算"任务（如加法）
   - Decoder-Only 适合纯生成任务（如语言模型）
   - 没有"最好"的架构，只有"最适合"的架构

2. 数据质量 > 数据数量 > 模型大小
   - 实验 1.3 混合位数训练 > 实验 1.2 单一位数训练
   - 实验 2.3 词级 Tokenizer > 字符级 + 大语料
   - 好的数据设计比盲目增加数据量更有效

3. Transformer 是"模式匹配"机器
   - 它能学会加法进位规则，但不是真正"理解"数学
   - 它能生成流畅文本，但只是复现训练语料模式
   - 这是强大之处，也是局限所在
```
